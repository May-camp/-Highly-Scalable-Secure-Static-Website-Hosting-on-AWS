🌐 Highly Scalable & Secure Static Website Hosting on AWS
📌 Executive Summary
This project demonstrates how to host a high-performance, secure, and low-latency static website using Amazon S3 as the storage origin and Amazon CloudFront as a global Content Delivery Network (CDN). The entire infrastructure is automated and provisioned as code using Terraform (AWS Provider v5.x).

To ensure cloud security best practices, direct public access to the S3 bucket is completely blocked. Content delivery is strictly gated through CloudFront using Origin Access Control (OAC), forcing all traffic to pass through AWS Edge Locations for caching and security filtering.

🏗️ Architecture Design & Caching Mechanism
When a recruiter asks: "Why use CloudFront instead of just making the S3 bucket public?" This step-by-step breakdown explains the exact business and technical value of this architecture:

Global Content Caching: Amazon CloudFront caches web assets (HTML, CSS, JavaScript, images) across hundreds of global Edge Locations.

First Request (Cache Miss): When a user requests a file from a far-off location, CloudFront fetches the original asset from the private S3 bucket, delivers it to the user, and caches a copy at the nearest Edge server.

Subsequent Requests (Cache Hit): When other users nearby request the same asset, CloudFront serves it instantly from the local cache.

The Result: * Ultra-low Latency: Drastically reduces network hop distances, speeding up page load times regardless of user location.

High Scalability: Offloads traffic from the S3 bucket. The infrastructure can seamlessly handle sudden viral traffic spikes without origin degradation.

Reduced Costs: Serving traffic from CloudFront edge caches is generally more cost-effective than constant read operations directly from an S3 bucket.

🛠️ Step-by-Step Technical Guide (Terraform Infrastructure)
Step 1: Provider Setup Configuration
We lock down the Terraform provider requirements to ensure stability across deployments by using modern features of the AWS v5.x provider.

Step 2: Provisioning the Private S3 Bucket (Origin)
The bucket is deployed entirely private. All public access configurations are explicitly blocked to prevent accidental data leaks.

Step 3: Configuring CloudFront Origin Access Control (OAC)
OAC is the modern AWS security standard (replacing the legacy OAI). It signs requests sent to the S3 bucket using AWS Signature Version 4 (SigV4), validating that requests are legitimately generated by our specific CloudFront distribution.

Step 4: Creating the CloudFront Distribution (CDN)
The distribution is configured to enforce security controls, route HTTP traffic safely to HTTPS, and map dependencies back to our OAC settings

Step 5: Enforcing the S3 Bucket Policy
We attach a deterministic bucket policy to grant read access (s3:GetObject) exclusively to the CloudFront Service Principal, constrained to our specific CloudFront distribution's ARN.
